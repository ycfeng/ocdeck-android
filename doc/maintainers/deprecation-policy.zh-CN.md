# 弃用策略

[English](deprecation-policy.md)

本文档是英文 canonical 弃用策略的便利翻译。中英文发生差异时，以英文文档为准。

OC Deck 仍处于 pre-1.0。接口和行为可以变化，但影响用户、持久化配置、受支持连接路径或公开工作流的变化，仍必须明确公告并谨慎迁移。

## 适用范围

本策略适用于：

- 用户可见功能和工作流。
- 持久化设置、服务器配置、凭据引用和本地数据格式。
- 受支持 Android 版本、ABI、OpenCode Server 兼容性和连接方式。
- 公开记录的构建、测试、发布和维护流程。
- 稳定 bridge API 与不可变 bridge 制品坐标。

对用户、集成方、持久化数据或文档都没有影响的内部实现细节，不要求公开弃用期。

## Pre-1.0 正常提前期

正常的 pre-1.0 弃用必须至少提前一个公开版本公告，之后才能移除或启用不兼容行为。在某个公开版本中弃用的功能，最早只能在下一个公开版本移除，除非明确承诺更长时间。

公告必须说明：

- 弃用内容与原因。
- 最早可能移除或产生不兼容行为的版本。
- 受影响对象及如何识别受影响状态。
- 迁移路径、替代方案、导出路径，或明确说明没有替代方案。
- 备份、回滚、服务端、凭据或签名连续性影响。

## 紧急例外

只有在延迟会实质加重安全漏洞、隐私暴露、法律或许可证违规、签名泄露、破坏性数据风险，或外部平台强制不兼容时，才允许立即移除或缩短提前期。

使用紧急例外时：

- 缩小受影响范围，不夹带无关 breaking change。
- 在不公开 exploit 细节或敏感材料的前提下说明例外原因。
- 提供可行范围内最安全的迁移、恢复或回滚说明。
- 在受影响版本的变更日志、发布说明和迁移指南中记录决定。

## 必需记录

每项公开弃用或移除都必须记录到所有适用位置：

- `CHANGELOG.md` 的 `Deprecated`、`Removed`、`Changed` 或 `Security`。
- `release-notes/vX.Y.Z.md` 的 `Breaking changes` 与 `Deprecated and removed`。
- 发布说明中的迁移章节，或其链接的版本化迁移文档。
- 当静默继续可能丢失数据、凭据、连接能力或访问权时，提供用户可见警告或校验。

Pull Request label 和 GitHub 自动说明只是补充，不能替代这些记录。

## 迁移安全

- 不得静默丢弃用户修改过的服务器配置、凭据、项目引用或草稿。
- 迁移必须区分历史自动生成默认值和用户修改值。
- 凭据轮换必须保留事务 alias 与清理规则；弃用不能成为降低 secret 处理要求的理由。
- 不得在已有 Maven 坐标下改变 bridge 字节。必须发布新的 `BRIDGE_VERSION` 并记录迁移。
- 破坏更新安装或签名连续性的变化必须接受明确的发布阻断审查。

## 移除审查

移除前应确认公告期已经结束、迁移说明仍正确、决策不依赖未授权遥测或用户数据、存在持久化旧状态时测试已覆盖其处理，而且全部文档链接都指向替代方案或最终状态。
